### Attacks & Vulnerabilities

[### You have got to be KDDI-ng – Japanese telco exposes 14.2 million managed email credentials (2 minute read)](https://www.theregister.com/cyber-crime/2026/06/24/you-have-got-to-be-kddi-ng-japanese-telco-exposes-142-million-managed-email-credentials/5260555?utm_source=tldrinfosec)

KDDI reported unauthorized access to its managed email platform on June 17 after attackers exploited a vulnerability in third‑party software, potentially exposing data for up to 14.2 million accounts. The service hosts email for KDDI and five Japanese ISPs. Passwords were hashed and encrypted, but affected users face phishing and identity‑theft risks while regulators and customers await the full scope of the incident.

[### XSS-to-Root Attack Delivers Malicious Code via Wi-Fi SSID or LoRa Node Name (2 minute read)](https://it4sec.substack.com/p/xss-to-root-attack-over-the-air-send?utm_source=tldrinfosec)

Security researcher Sasha Romijn demonstrated an XSS attack vector that delivers malicious code over the air through Wi-Fi SSIDs or LoRa node names, exploiting infrastructure web interfaces that render these network identifiers without proper sanitization. The vulnerability is especially dangerous in environments that rely on outdated embedded browsers with infrequent security updates, where a successful injection can escalate from script execution to root access on the underlying device. Defenders should audit management interfaces that display SSID or node name fields, apply input sanitization and output encoding to those components, and update embedded browsers to current versions where patching is available.

[### In Less Than 24 Hours, Attackers Weaponize Cisco CUCM Flaw (2 minute read)](https://www.darkreading.com/cyberattacks-data-breaches/less-than-24-hours-attackers-weaponize-cisco-cucm-flaw?utm_source=tldrinfosec)

Attackers are exploiting CVE-2026-20230 in Cisco Unified Communications Manager to gain root access through an unauthenticated SSRF path in the WebDialer service. The exploit chain uses crafted HTTP requests, a rogue Apache Axis service, and staged JSP web shells to reach command execution and privilege escalation. Defused saw weaponization within 24 hours of public PoC release, and Horizon3.ai published a test while urging rapid patching or WebDialer shutdown on unpatched systems.

### Strategies & Tactics

[### What Happened After 2,000 People Tried to Hack My AI Assistant (3 minute read)](https://simonwillison.net/2026/Jun/26/hack-my-ai-assistant/#atom-everything?utm_source=tldrinfosec)

Fernando Irarrázaval launched hackmyclaw.com to test whether his OpenClaw AI assistant could be tricked into leaking a secrets.env file via email, drawing over 2,000 participants who sent more than 6,000 prompt-injection-style messages. The secret was never successfully exfiltrated, indicating that the anti-prompt-injection measures were effective, though the exercise incurred approximately $500 in token costs and triggered a suspension of the associated Google Gmail account. Defenders running similar red-team exercises should pre-provision API budgets, isolate mail accounts from production services, and treat provider suspension as a likely operational side effect of high-volume adversarial testing.

[### Introducing Package Proxy: Supply-Chain Safety Checks Without Client-Side Software (4 minute read)](https://blog.thinkst.com/2026/06/introducing-package-proxy-supply-chain-safety-checks-without-client-side-software.html?utm_source=tldrinfosec)

In the wake of numerous recent supply chain compromises, Thinkst released Package Proxy as an inline package guard. Package Proxy supports dependency cooldown, upload-mechanism regression checks, a bypass for explicit audit-fix steps, and block/allow lists. Package Proxy deploys on Cloudflare, uses Workers to serve packages, and uses a D1 Database for logging, enabling SQL queries into package installs.

[### Goblin Loot: Proxmox and Adversaries (4 minute read)](https://www.goblinloot.net/2026/06/adversaries-in-proxmox.html?utm_source=tldrinfosec)

As Proxmox displaces VMware in many shops, this detection-engineering walkthrough outlines how adversaries operate on PVE nodes: enumerating realms via an unauthenticated GET to /api2/json/access/domains/, brute-forcing accounts, then living off the land with pvesh to enumerate or destroy guest VMs and abusing IP sets in /etc/pve/firewall/cluster.fw to lock responders out. pvesh activity never reaches the pveproxy audit trail and is visible only through process execution and journald, so defenders should model normal PVE CLI usage by SSH-session lineage, command diversity, and invocation directory, and deploy AuditD watches keyed to executables under /usr/sbin and /usr/bin (qm, pct, pvecm, pvesm, pveum, and pvesh) plus writes to /etc/pve/ and /usr/share/perl5/PVE/. Collect the pveproxy access log, along with the named systemd units (pvedaemon, pve-cluster, pveproxy, pvefw-logger, proxmox-firewall, and related services), via journald to close the gaps left by GUI-routed and cluster tasks.

### Miscellaneous

[### The "Akrites" vulnerability-mitigation project launches (1 minute read)](https://lwn.net/Articles/1079657/?utm_source=tldrinfosec)

The Linux Foundation and a broad coalition of co-signers launched Akrites, a project to fast-track upstream vulnerability fixes and prioritize patch deployment over public disclosure, treating undisclosed flaws in widely deployed packages as weapons, keeping fixes confidential until deployed, and acting as the maintainer of last resort for unmaintained critical packages.

[### Amazon Q Developer Flaw Could Let Malicious Repos Run Code via MCP Configs (2 minute read)](https://thehackernews.com/2026/06/amazon-q-developer-flaw-could-let.html?utm_source=tldrinfosec)

A bug in Amazon Q Developer let MCP configs in a cloned repo start local servers that inherited live AWS credentials, enabling command execution without extra login. Wiz showed config-driven theft of the active AWS session, and AWS patched the issue in Language Servers for AWS, requiring updated IDE plugins and adding explicit prompts before untrusted MCP servers run.

[### NSA's SIGINT Enabling Project Pushes Weakened TLS Post-Quantum Standard (1 minute read)](https://nsa.2026.action.cr.yp.to/)

The NSA's SIGINT Enabling Project is reportedly pressuring the IETF to adopt "ietf-tls-mlkem" as a replacement for "ietf-tls-ecdhe-mlkem," a shift the author characterizes as a deliberate weakening of post-quantum TLS cryptography. NSA advocates allegedly manipulated the IETF TLS working group's voting process to secure a favorable outcome, fitting a historical pattern of the agency sabotaging cryptographic standards through standards bodies. Readers concerned about protocol integrity are urged to submit opposition comments to the IETF mailing list by July 7, though the claims rest on a single advocacy source with an evident stake in the outcome.
